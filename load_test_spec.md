# 負荷テスト仕様書 (Load Testing Specification)

本仕様書は、リアルタイム投票・ランキングシステム（SANRENTAN PRO）をRender上で公開・運用するにあたり、システムが想定される同時接続数とトラフィックに耐えられるかを検証するための負荷テスト仕様を定義したものです。

---

## 1. テスト目的
- イベントやゲーム大会でのリアルタイム利用を想定し、同時接続数に対するWebソケット通信の安定性を検証する。
- 参加者（Voter）による投票処理（`submit-vote`）およびランキング要求（`get-ranking`）のラウンドトリップ時間（RTT）が実用範囲内（平均200ms以下）に収まるかを確認する。
- Renderのフリープランまたは有料プランにおける限界接続容量（Concurrent Connections）のボトルネックを把握する。

---

## 2. テスト対象システム構成
- **フロントエンド**: React (Vite SPA) -> Expressサーバーから静的ファイル配信 (`dist/index.html`)
- **バックエンド**: Node.js (Express + Socket.io)
- **データベース**: ローカルJSONファイル (`db.json`) による同期I/O処理
- **デプロイ先**: Render Web Service (Node.js環境)
  - URL形式: `https://[サービス名].onrender.com`

---

## 3. 負荷テスト環境
### 3.1. サーバー環境（Render）
- **プラン候補**:
  - **Freeプラン**: 0.1 CPU, 512 MB RAM (アイドル15分でスピンダウンあり、ディスクは揮発性)
  - **Starter/Standardプラン（推奨）**: より安定したCPU/RAMリソース、永続ディスクの追加が可能
- **データ永続性**:
  - デフォルトでは `db.json` はエフェメラル（再起動で初期化）となります。テスト実行中は問題ありませんが、本番運用時に投票データを永続化したい場合は Persistent Disk のアタッチを推奨します。

### 3.2. クライアント環境（テスト実行機）
- **ツール**: Locust (Python 3.x)
- **必要ライブラリ**:
  - `locust` (負荷テストエンジン)
  - `python-socketio` (Socket.ioプロトコル通信用クライアント)
  - `websocket-client` (WebSocketトランスポート用)

---

## 4. 負荷テストシナリオ
Locustスクリプト (`locustfile.py`) にて、以下の2種類のユーザーモデルを重み付け（比率）を変えて並行実行します。

### 4.1. 参加者モデル (`VoterUser`): 比率 80% (Weight: 5)
一般参加者の端末を模倣し、接続を維持しながらランダムな間隔（2〜5秒）で以下の操作を繰り返します。
1. **投票送信 (`submit-vote`)** (タスク比重: 3)
   - 選択肢（ランナーA〜E）から重複なく3名を選択し投票を送信。
   - 送信からサーバー応答（`vote-success`）が届くまでの時間をRTTとして計測（タイムアウト5秒）。
2. **ランキング取得 (`get-ranking`)** (タスク比重: 2)
   - 最新のランキングデータを要求。
   - 要求送信からデータ受信（`ranking-data`）までの時間をRTTとして計測（タイムアウト5秒）。

### 4.2. モニター画面モデル (`MonitorUser`): 比率 20% (Weight: 1)
会場の大型スクリーンなどを模倣し、接続を常時維持します。
- **アイドル受信 (`idle`)**
   - 能動的な要求は行わず、サーバーからブロードキャストされるイベント（`sync-state` / `vote-count-update` / `results-broadcast` など）を受信して、応答時間をロギングします。

---

## 5. 測定指標 (KPI) と目標値

| 測定項目 | 目標値 (Target) | 備考 |
| :--- | :--- | :--- |
| **同時接続数 (Target Users)** | **100 〜 300** 同時接続 | イベント規模に応じた同時接続テスト |
| **接続成功率 (Success Rate)** | **99.9% 以上** | エラー率 0.1% 以下 |
| **投票応答時間 (RTT)** | **平均 200ms 以下** (95%ile 500ms 以下) | `submit-vote` -> `vote-success` |
| **ランキング応答時間 (RTT)** | **平均 200ms 以下** (95%ile 500ms 以下) | `get-ranking` -> `ranking-data` |

---

## 6. テスト実施手順

### 6.1. 準備段階 (クライアント側)
テスト実行マシンにPython 3環境を準備し、以下のコマンドで必要な依存関係をインストールします。

```bash
pip install locust python-socketio websocket-client
```

### 6.2. サーバーの確認
Renderにアプリがデプロイされ、正常起動していることをブラウザで確認します。
※フリープランの場合、起動直後はコールドスタートのため初回接続に数十秒かかる場合があります。テスト開始前に必ずWebページにアクセスしてアクティブ状態にしてください。

### 6.3. Locustの起動
プロジェクトのルートディレクトリで以下のコマンドを実行します。

```bash
locust -f locustfile.py
```

### 6.4. テストパラメーターの設定
1. ブラウザで `http://localhost:8089` にアクセスします。
2. 以下の項目を設定して **「Start swarming」** をクリックします。
   - **Number of users (Peak concurrency)**: 最大同時ユーザー数（例: `100`）
   - **Spawn rate (Users started/second)**: 1秒あたりのユーザー増加数（例: `5` 〜 `10`。Socket.ioの接続バーストを防ぐため緩やかな増加を推奨）
   - **Host**: テスト対象のURL (例: `https://sanrentan-pro.onrender.com` またはローカルテスト時は `http://localhost:3001`)

### 6.5. 統計の分析と終了
- **Chartsタブ**で、同時接続数に対する「応答時間 (Response Times)」および「秒間リクエスト数 (RPS)」の推移グラフを確認します。
- 応答時間が急激に悪化し始めるユーザー数（限界値）を記録します。
- 目標時間（例: 5〜10分間）に達したら、画面上部の **「Stop」** ボタンを押して負荷テストを停止します。

---

## 7. Render固有の留意事項
- **CPU/メモリ枯渇リスク**:
  Express+Socket.ioはシングルスレッド（Node.js）で動作します。イベントループが `db.json` のファイルI/Oでブロックされると、同時接続数が多い場合（150名以上など）に応答時間が極端に低下する可能性があります。その場合は、有料プランへのスケールアップやデータベース（SQLite/PostgreSQL等）の使用を検討してください。
- **タイムアウト設定**:
  WebSocketの接続が切断されないよう、Renderのプロキシ仕様に合わせて適切なハートビートが行われることを確認してください（`python-socketio` はデフォルトで自動ハンドシェイクを維持します）。
